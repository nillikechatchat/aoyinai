## 引言

黑客松的核心不是"写最多的代码"，而是"在有限时间内解决一个真实问题"。

本文基于多个获奖项目的复盘，总结从问题定义到演示准备的完整方法论。

## 一、问题定义（前 2 小时）

### 1.1 选题原则

**好问题的特征**：
- **具体**：不是"改善教育"，而是"帮高三学生整理错题"
- **可验证**：有明确的成功标准（能演示、能量化）
- **有价值**：解决后有人会用、会付费
- **可实现**：48 小时内能做出 MVP

**差问题的特征**：
- 太大："用 AI 改变世界"
- 太模糊："提升用户体验"
- 没有用户："技术很酷但没人需要"
- 做不完："完整的产品"

### 1.2 问题拆解

```
原始问题：帮农村老人用方言看病
  ↓ 拆解
子问题 1：老人说方言，医生听不懂
子问题 2：医学术语老人听不懂
子问题 3：老人不知道该看什么科
  ↓ 选择
聚焦子问题 1：方言 → 普通话医疗翻译
```

### 1.3 用户画像

```
用户：65 岁农村老人，只会说方言
场景：去县城医院看病
痛点：说不清楚症状，医生听不懂方言
现有方案：让子女陪同（但子女不在身边）
期望：用方言说症状，医生能听懂
```

## 二、方案设计（2-4 小时）

### 2.1 MVP 定义

**MVP 原则**：只做 1 个核心功能，做到能演示。

```
完整方案：方言 → 普通话翻译 + 症状识别 + 科室推荐 + 医生解释翻译
  ↓ 砍功能
MVP：方言 → 普通话翻译（只做语音转文字 + 翻译）
```

### 2.2 技术选型

**选型原则**：用最熟悉的技术，不学新技术。

| 需求 | 选项 | 推荐 |
|------|------|------|
| 前端 | React / Vue / 原生 HTML | 最熟悉的 |
| 后端 | Node.js / Python / Go | 最熟悉的 |
| 数据库 | SQLite / JSON 文件 | MVP 用 JSON |
| AI 模型 | GPT-4o / Claude / 开源 | API 最快的 |
| 部署 | Vercel / 本地 | 最快的 |

### 2.3 架构设计

```
用户输入（方言语音）
    ↓
语音识别（Whisper API）
    ↓
方言文本
    ↓
翻译（LLM）
    ↓
普通话文本
    ↓
展示给医生
```

**关键决策**：不要自己训练模型，用现成的 API。

## 三、实现阶段（4-36 小时）

### 3.1 时间分配

```
第 1 天（0-24 小时）：
  0-2h：问题定义 + 方案设计
  2-8h：核心功能开发
  8-12h：集成调试
  12-16h：休息
  16-20h：功能完善
  20-24h：Bug 修复

第 2 天（24-48 小时）：
  24-30h：功能优化
  30-36h：演示准备
  36-40h：排练
  40-44h：最终调整
  44-48h：提交
```

### 3.2 开发策略

**并行开发**：
- 前端和后端同时开发
- 使用 mock 数据让前端先跑通
- 后端 API 好了再对接

**先跑通再优化**：
```python
# 第一步：硬编码，跑通流程
def translate(text):
    return "这是翻译结果"

# 第二步：接 API
def translate(text):
    response = openai.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": f"翻译：{text}"}]
    )
    return response.choices[0].message.content

# 第三步：优化 prompt
def translate(text):
    prompt = f"""你是一个医疗翻译专家。请将以下方言翻译成标准普通话，保留医学含义：
    
方言：{text}
普通话："""
    ...
```

### 3.3 避坑清单

- **不要在比赛开始后才讨论"做什么"**：提前想好 2-3 个方向
- **不要试图用新技术**：用最熟悉的技术栈
- **不要熬夜到天亮**：疲劳状态下写的代码全是 bug
- **不要忽略错误处理**：演示时出错很尴尬
- **不要做完美主义**：MVP 能演示就行

## 四、演示准备（最后 8 小时）

### 4.1 演示脚本

```
1. 问题（1 分钟）
   "我妈妈是农村老人，只会说方言。上次她去看病，医生听不懂她说什么。"

2. 解决方案（1 分钟）
   "我们做了一个方言医疗翻译 App，老人用方言说症状，医生看到普通话翻译。"

3. 现场演示（2 分钟）
   - 打开 App
   - 用方言说一句话
   - 展示翻译结果
   - 展示关键功能

4. 技术亮点（30 秒）
   "我们用了 Whisper 做语音识别，GPT-4o 做翻译，延迟控制在 2 秒内。"

5. 总结（30 秒）
   "这个产品帮农村老人跨越就医的语言障碍。"
```

### 4.2 演示技巧

- **现场演示**：比 PPT 截图有说服力 100 倍
- **准备 Plan B**：如果现场演示失败，准备录屏视频
- **讲故事**：评委想听"你解决了什么问题"，不是"你用了什么技术"
- **控制时间**：超时是最大的扣分项

### 4.3 提交清单

- [ ] 代码提交到 GitHub
- [ ] README 包含项目说明、运行方法、技术栈
- [ ] 演示视频（备份）
- [ ] PPT（如果需要）
- [ ] 按要求格式提交

## 五、赛后复盘

比赛结束后，无论名次如何：

1. **记录学到的东西**：技术、方法、协作
2. **收集反馈**：评委的意见、其他团队的方案
3. **开源代码**：为简历加分
4. **保持联系**：和队友、其他参赛者保持联系

## 总结

黑客松的核心是"在有限时间内解决一个真实问题"。技术只是手段，解决问题才是目的。

**关键原则**：
- 问题定义比技术实现更重要
- MVP 比完整产品更有效
- 演示准备和代码质量同等重要
- 团队协作比个人能力更关键

> 本文方法论适用于 24-48 小时的黑客松，具体时间分配可根据赛制调整。
